Arch Linux下Opencv3.4在Pyenv虚拟环境编译教程

2020-05-07
Opencv安装 / 机器学习
字数统计: 2.6k字   |   阅读时长≈ 12分钟

Opencv在3.4.0之后版本放弃支持使用者训练自己的Haar联级分类器，为了使用这个已经被官方抛弃的好用功能，记录一下相关环境的构造过程。

Pyenv安装和虚拟环境配置

  • 查看Github链接即可，需要安装pyenv和pyenv-virtualenv

    pyenv
    pyenv-virtualenv

本机环境

  • 注意虚拟环境要使用pip install numpy命令安装numpy

    Python 3.5.0 虚拟环境
    OpenCV 3.4.0

Opencv安装

1. 安装所需依赖

  • Arch Linux:

    sudo pacman -S opencv vtk hdfs5 -y

  • Ubuntu:

    sudo apt install build-essential cmake pkg-config
    sudo apt-get install libavcodec-dev libavformat-dev libswscale-dev libv4l-dev libxvidcore-dev libx264-dev libatlas-base-dev gfortran

    2. 下载opencv和opencv_contrib对应版本压缩包

  • 想安装别的版本更改链接为对应的版本号即可

3. 预编译

  • 使用cmake, 更多参数请参考另一篇文章，注意PYTHON3_EXCUTABLE、PYTHON3_INCLUDE_DIR、PYTHON3_LIBRARY，这几个参数是编译到虚拟环境的关键

    4. 编译过程

    • 使用make进行编译

      make -j8

    • 编译完成后进行安装

      sudo make install

5. 正常安装结果

  • 一切过程正常后需要验证一下cv2.so文件是否已经正常生成，没有去看后面疑难杂症部分

6. 后续完善至Python正常导入

  • 进入编译好的文件的目录，使用ln命令软链接到虚拟环境目录

6. 特别提醒：OpenCV卸载方法

  • 进入build文件夹，使用如下命令

    sudo make uninstall

安装Opencv遇到的疑难杂症

1. 因网络问题无法下载某些文件中止cmake编译

  • 错误描述

  • 根据提示打开CMakeDownloadLog.txt

  • 解决方法(配置proxychains通过http代理链接外网并使用wget下载文件，没有配置去掉proxychains)

    • 下载完成后可以在文件夹下找到该文件
    • 重命名为”md5-文件名”，放在 /.cache/dnn_face_detector/ 文件夹下面，或者直接放到目标路径 /samples/dnn/face_detector/（不用重命名）。

2. 致命错误：dynlink_nvcuvid.h：没有那个文件或目录

  • 错误描述

  • 产生原因

    cmake编译时会自动加入CUDA测试，但3.4.0版本opencv只支持版本<= 6.5的CUDA，截止2020年5月，CUDA已经更新10.0版本，CUDA的API在编码上的更新已经不再支持opencv，需要对cmake参数中的CUDA相关参数进行设置。

  • 解决方法

    • cmake命令中增加以下参数

      -D BUILD_opencv_cudacodec=OFF
      -D ENABLE_PRECOMPILED_HEADERS=OFF

3. 致命错误：Python.h：没有那个文件或目录

  • 错误描述

  • 产生原因
    Python.h位于Python环境中的include目录下，用于编译时遍历部分文件，寻找不到Python.h即cmake预编译时PYTHON3_INCLUDE_DIR参数设置错误，需要进行需改。

  • 解决方法

    1. 已经安装Python环境仍然报错，修改cmake命令中的PYTHON3_INCLUDE_DIR参数，一定要使用/home/user这种绝对路径，~/opencv/这种的相对路径会导致找不到include路径。
    2. 没有安装Python换进或者环境不完整，重新安装。

4. 错误：‘class vtkPolyDataMapper’

  • 错误描述
  • 产生原因
    Arch Linux具有滚动更新的特性，本机vtk版本为8.2.0，此版本已经将vtkMapper的ImmediateModeRenderingOff()方法进行了移除。
  • 解决方法
    • 方案1：进入对应文件将报错的语句进行注释。
    • 方案2：对vtk进行降级之后再次编译

5. 没有规则可制作目标“/usr/lib/libGLEW.so”

  • 错误描述

  • 产生原因
    一般大部分原因是因为make 编译的时候，动态链接库的链接性中断，本机中也发现并不存在libGLEW.so这一文件，推测应该与相关软件更新导致的不兼容有关。

  • 解决方法

    • cmake命令中增加以下参数禁止编译相关功能

      -D BUILD_opencv_viz=OFF \

6. 错误：invalid conversion from ‘const char’ to ‘char’ [-fpermissive]

  • 错误描述

  • 解决方法

    • 找到报错的地方，将原句

      char* str = PyString_AsString(obj);

      修改为

      const char* str = PyString_AsString(obj);

7. cmake预编译没有找到Python3的库，全过程没有报错，但是编译安装完成后找不到cv2.cpython-35m-x86_64-linux-gnu.so

  • 错误描述（cmake预编译时的显示结果）

  • 产生原因

    当前环境找不到Python3的库

  • 解决方法

    首先make clean，返回上层目录，删除build目录（必须要删除再创建），进入pyenv虚拟环境再使用cmake命令，得到如下显示结果：
